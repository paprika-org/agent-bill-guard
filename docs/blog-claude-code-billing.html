Claude Code Billing: Track Token Costs Locally with agent-bill-guard

Published May 12, 2026

Claude code billing gets expensive faster than many developers expect because long prompts, repeated tool calls, and retry loops compound across a session. If you are running autonomous or semi-autonomous workflows, you need per-request visibility instead of waiting for a provider dashboard to tell you the damage afterward. agent-bill-guard is a small local Python proxy that sits between Claude Code and the Anthropic API so you can inspect token usage, cost, and tool activity in real time.

Why Claude Code Billing Becomes Hard to Predict

The main problem is not just prompt size. Claude Code style workflows often include a growing conversation buffer, structured tool schemas, command output, file diffs, and retries after tool failures. A single task can fan out into many requests, and each request may carry more context than the last. From the terminal, that usually looks like “the agent is still working,” but the bill may be rising on every turn.

This is why raw invoice totals are not enough for debugging cost. To understand claude code billing, you need request-level attribution: input tokens, output tokens, estimated cost, and whether tool-heavy turns are the real driver. Without that, optimization turns into guesswork.

Local instrumentation is especially useful when you are tuning prompts, evaluating agent loops, or testing tools that can accidentally create recursive or overly chatty behavior.

What agent-bill-guard Actually Does

agent-bill-guard is intentionally narrow in scope. It proxies API traffic locally, prints usage details to the terminal, and can enforce configurable spend limits. That makes it practical for development machines, CI experiments, or any place you want a lightweight safety rail without adding another service.

Its design is appealing for developer environments because it is stdlib-only Python. There is no dependency stack to audit, no database, and no hosted dashboard. You run it locally, point Claude Code at the proxy, and inspect the output as requests pass through.

git clone https://github.com/paprika-org/agent-bill-guard.git
cd agent-bill-guard

# Start the local proxy
python3 agent_bill_guard.py --listen 127.0.0.1:8080 --upstream https://api.anthropic.com

# Example terminal output
[req 14] model=claude input_tokens=18234 output_tokens=911
[req 14] tools=bash,read_file estimated_cost_usd=0.47
[session] total_requests=14 total_cost_usd=3.92
[guard] soft_limit_usd=5.00 hard_limit_usd=7.50

# When hard limit is reached, the proxy blocks new requests
[guard] request denied: session spend limit exceeded

The value here is operational, not cosmetic. You can see whether an expensive turn came from prompt growth, output verbosity, or an agent repeatedly invoking tools. For claude code billing, that feedback loop is much more actionable than reading aggregate usage later.

How to Use It for Cost Debugging

A good workflow is to treat cost tracing like performance tracing. Start with a known task, run it through the proxy, and observe which requests dominate spend. In many cases, the highest-cost turns are not the ones you expect. A harmless-looking refactor can become expensive if the agent keeps re-reading large files or embedding prior command output in full.

Once you have the request breakdown, you can make targeted changes:

  • Reduce oversized context windows by trimming file reads and irrelevant history.
  • Lower retry churn by fixing flaky tool commands or adding clearer failure handling.
  • Set session-level limits for experiments so an agent stops before a loop becomes costly.
  • Compare tasks before and after prompt changes instead of relying on intuition.

This is also where honest limitations matter. agent-bill-guard can estimate and track usage in real time, but it does not replace provider-side billing records. If Anthropic pricing changes, if a model has special accounting rules, or if you need formal invoice reconciliation, you still need the upstream source of truth. The proxy is for local visibility and guardrails, not financial reporting.

Common Mistakes

Forgetting that context growth is cumulative. Developers often test a single request and conclude the cost is acceptable. In real agent sessions, repeated file snapshots, command output, and conversation history make later requests far more expensive than the first one.

Using spend limits only after problems appear. If you enable hard limits after debugging starts, you are already exposed to the runaway case you meant to prevent. Set thresholds before experiments, especially when testing new tools or autonomous flows.

Assuming tool calls are “free overhead.” The tool invocation itself is not the whole story. The request may include tool schemas, prior results, and extra recovery prompts, all of which affect claude code billing.

Expecting historical analytics. agent-bill-guard is a local runtime proxy. If you want cross-machine aggregation, long-term dashboards, or team-wide reporting, you will need to build that separately.

Frequently Asked Questions

Does agent-bill-guard change Claude Code prompts or responses?

No. It acts as a local proxy for request and response inspection. It tracks usage, cost, and tool calls, but it does not rewrite prompts or alter model output.

Can it stop a runaway session automatically?

Yes. You can configure spend limits so requests are blocked once a threshold is reached, which is useful when an agent starts looping or making excessive tool calls.

Is agent-bill-guard a hosted billing dashboard?

No. It runs locally, uses only the Python standard library, and prints per-request details directly in the terminal instead of sending usage data to a remote service.

What does it not solve for Claude Code billing?

It does not replace Anthropic account billing, historical invoice reporting, or pricing policy changes. It helps with local visibility and guardrails during development sessions.
